## 二、核心问题清单

### P0 — 必须修复（阻碍可用性）

#### P0.1 知识库太小

22篇笔记覆盖钱博士总输出（685直播+443短视频）的 **~3%**。一个 RAG 系统的实际价值取决于知识库的广度和深度。当前 22 篇笔记中仅 5 篇来自直播（每篇是完整2h的核心提炼），17 篇来自 1-5 分钟短视频。对于"投资顾问"级别的查询需求，这个知识量完全不夠。

**影响**: 任何涉及长期趋势判断、跨板块对比的查询，RAG 返回的都是"能找到的最相关的这一段"，不一定是最有用的那一段。

#### P0.2 脚本和数据散落在三个位置，路径硬编码

- `build_vector_db.py` 硬编码 `obsidian_path = "E:/obsidian-vault/学习/钱博士"`
- `query_rag.py` 用 `Path(__file__).parent / ".." / "vector_db"` 找向量库
- `portfolio.py` / `tracking.py` 用 `Path(__file__).parent.parent / "portfolio.json"`
- `daily_brief.py` 读环境变量 `QIABOSHI_OBSIDIAN_PATH`

**结果**: 换机器/换用户/换目录结构，所有路径全断。迁移成本极高。

#### P0.3 无增量更新，每次重建全库

`build_vector_db.py` 只有 `--rebuild` 模式：

```
删除旧库 → 重新扫描全部笔记 → 重新embedding → 重新写入
```

当笔记从 22 篇增长到 100+ 篇时，重建耗时从 1 分钟增长到 10+ 分钟。更关键的是：**这阻断了持续积累**。加一篇新笔记的心理成本太高了。

### P1 — 应该修复（影响体验）

#### P1.1 反馈闭环不存在

设计文档花了大量篇幅描述"进化反馈层"（短线/中线/长线三种反馈回路、逻辑验证机制、改进日志），但**代码里一行都没实现**。

没有：
- 钱博士观点 vs 市场实际走势的对照表
- 买卖盈亏的汇总分析
- "这个观点后来被打脸了"的检测机制

当前系统在**信息搬运**层面，没有到**辅助决策**。

#### P1.2 盘前简报不含任何实时行情

`daily_brief.py` 生成的简报完全是 RAG 文本：

```
【当前大势】 → RAG查询"当前市场环境 牛市 熊市"的Top-2
【重点板块】 → RAG查询"看好的板块 确定性最高"的Top-2
【风险提醒】 → RAG查询"风险提示 逃顶信号"的Top-2
```

没有指数数据、没有板块涨跌、没有持仓盈亏——尽管 `tracking.py` 和 `portfolio.py` 能提供这些数据，`daily_brief.py` **完全没调用它们**。

#### P1.3 yfinance 调用效率低

`tracking.py` 对每个标的单独调 `yf.Ticker()`，20 次独立 HTTP 请求，耗时 15-30 秒。且没有：
- 批量下载（`yf.download()`）
- 缓存层（收盘后前次调用和下次调用的结果一样）
- 非交易时段优雅降级

#### P1.4 无依赖声明

没有 `requirements.txt` 或 `pyproject.toml`。新环境需要手动 pip 一个个装。`chromadb`、`yfinance`、`requests` 等依赖的版本不锁定，未来可能因升级而破坏。

### P2 — 值得改进（长期完善）

#### P2.1 `generate_advice()` 概念有缺陷

`generate_advice()` 用硬编码查询字符串（如 "科技调整 要不要减仓"）做 RAG 检索，把 Top-1 结果当"投资建议"。但 RAG 相似度 ≠ 建议质量——高相似度匹配可能是完全不同语境下的片段。

**更好的方式**: 将 RAG 结果作为参考素材，结合分析框架做结构化判断，而不是简单搬运。

#### P2.2 美股涨幅榜只有 26 只预设标的

yfinance 1.5.1 没有 Screener API，所以 `tracking.py gainers` 只能扫描 26 只预设热门美股。真正的全市场异动（如新上市小票爆发）永远不会被发现。

#### P2.3 数据格式无版本号

`portfolio.json` / `tracking_pool.json` / `trades.json` 都没有版本字段。未来格式变更时，旧数据无法自动迁移。

## 三、基本原则（不可违背）

这些原则决定"什么该做、什么不该做、什么需要先讨论"。

### 原则1：增量优先，重建备选

**所有数据操作必须支持增量模式。** 重建（`--rebuild`）可以作为维护工具存在，但不能是唯一路径。

- RAG 库: 支持 upsert（按文档ID增/改/删）
- 持仓/标的池: 新增字段时兼容旧格式
- 转录文本: 新文件不覆盖旧文件

### 原则2：路径抽象，一处配置

**所有硬编码路径必须消灭。** 统一通过 `config.yaml` 配置。

- 脚本内不出现 `E:/obsidian-vault/...` 或 `Path(__file__).parent.parent/...`
- 所有路径引用走配置层
- 配置层提供默认值 + 环境变量覆盖 + 命令行覆盖三层机制

### 原则3：数据与代码分离

**数据文件（portfolio.json / tracking_pool.json / trades.json / 向量库）不应该和脚本在同一目录。**

- 脚本 → 位置无关，可被任何目录的代码 import
- 数据 → 集中在配置指定的数据目录下
- 向量库 → 数据目录下一个子目录

### 原则4：模块各自独立，通过CLI组合

**每个脚本做一件事，做好。** 组合通过 CLI 管道或 Hermes cron 编排，不通过脚本间 import 耦合。

```
portfolio.py  → 持仓管理 + 行情查询
tracking.py   → 标的池管理 + 行情扫描
query_rag.py  → RAG检索（纯查询，不依赖外部数据文件）
daily_brief.py → 整合所有模块的输出 + RAG → 简报 + 推送
```

例外: `query_rag.py` 作为纯查询引擎可以被其他脚本 import（目前已做到）。

### 原则5：向后兼容，版本化数据

**任何数据格式变更必须有自动迁移脚本。**

- 所有 JSON 数据文件首字段必须是 `"version": N`
- 脚本读取时检查版本号，不支持的版本报错提示迁移
- 迁移脚本 `scripts/migrate.py` 处理旧→新格式

### 原则6：反馈闭环要有代码落地

**设计文档说到的反馈机制，必须在一两个迭代内进入代码。**

- 逻辑验证表: `validation_log.json`，每期简报自动追加一条
- 进化日志: `CHANGELOG.md`，每次修改脚本自动追加（或至少手动记）
- 不要"等架构完善了再实现"——MVP 优先

## 四、目标架构

### 4.1 数据流目标

```
B站直播/视频
  → yt-dlp 下载（scripts/download.py）
    → GPU Whisper 转写（scripts/transcribe.py）
      → ASR 纠错（scripts/transcribe.py 内联）
        → LLM 结构化 → Obsidian笔记
          → RAG upsert（scripts/build_vector_db.py --incremental）
                    ↓
               ChromaDB  ← 查询 (query_rag.py)
                              ↕
  行情接口 ──→ tracking.py ──┐
                              ├──→ daily_brief.py ──→ 飞书/web
  持仓管理 ──→ portfolio.py ──┘
                              └──→ validation.py ──→ validation_log.json
```

### 4.2 目录结构目标

```
qianboshi-agent/
├── config.yaml                 ← 统一配置（核心！）
├── requirements.txt            ← 依赖声明
├── CHANGELOG.md                ← 进化日志
├── 架构文档.md                  ← 本文档
│
├── scripts/
│   ├── query_rag.py            ← RAG查询（纯查询引擎）
│   ├── build_vector_db.py      ← 向量库构建（支持增量upsert）
│   ├── daily_brief.py          ← 盘前简报（整合所有模块）
│   ├── portfolio.py            ← 持仓管理
│   ├── tracking.py             ← 标的池管理
│   ├── download_onnx.py        ← ONNX模型下载（断点续传）
│   ├── transcribe.py           ← 下载+转写流水线
│   ├── validate.py             ← 逻辑验证（新增）
│   └── migrate.py              ← 数据迁移（新增）
│
├── data/                       ← 所有数据集中在此
│   ├── portfolio.json
│   ├── trades.json
│   ├── tracking_pool.json
│   ├── validation_log.json     ← 新增：逻辑验证记录
│   └── vector_db/              ← 向量库（ChromaDB）
│
├── config.yaml                 ← 统一配置
│
└── [其他文件仍留在原位，通过迁移脚本逐步搬迁]
```

### 4.3 配置层设计

```yaml
# config.yaml 核心结构（示例）
version: 1

paths:
  obsidian_vault: E:/obsidian-vault/学习/钱博士
  data_dir: ./data                    # 相对路径时相对于项目根
  audio_dir: ./audio
  transcripts_dir: ./transcripts

rag:
  vector_db_path: ./data/vector_db    # 覆盖默认
  collection_name: qianboshi
  top_k: 5
  score_weights:
    similarity: 0.6
    freshness: 0.2
    type_boost: 0.2

market:
  proxy: http://127.0.0.1:7897
  cache_ttl_minutes: 30              # 行情缓存时间

notifications:
  feishu_webhook: ""                  # 从环境变量取，不硬编码
```

**优先级**: 代码默认值 < config.yaml < 环境变量 < 命令行参数

## 五、数据层设计

### 5.1 数据文件格式规范

所有 JSON 数据文件统一格式：

```json
{
  "version": 1,
  "updated": "2026-07-09",
  "data": { ... }
}
```

`version` 字段是必须的。这是向后兼容的第一道防线。

### 5.2 portfolio.json v2

```json
{
  "version": 2,
  "updated": "2026-07-09",
  "holdings": {
    "159992": {
      "shares": 1180,
      "avg_cost": 0.847,
      "name": "创新药ETF",
      "sector": "创新药",
      "buy_date": "2026-07-08"
    }
  },
  "cash": 0,
  "currency": "CNY",
  "total_cost": 999.46,
  "history": []
}
```

**变更**: 添加 `version` + `sector` + `buy_date` + `total_cost` 计算字段。**迁移脚本**自动从 v1（无 version 字段）升级。

### 5.3 trades.json v1

```json
{
  "version": 1,
  "trades": []
}
```

保持简洁。每条交易记录:

```json
{
  "id": "a1b2c3d4",
  "type": "buy",
  "code": "159992",
  "amount": 1180,
  "price": 0.847,
  "date": "2026-07-08",
  "time": "2026-07-08T09:30:00",
  "note": "首次建仓"
}
```

**变更**: 添加 `id`（UUID短前缀）和 `note` 字段。

### 5.4 tracking_pool.json v2

```json
{
  "version": 2,
  "updated": "2026-07-09",
  "stocks": { ... },      // 同现有格式
  "etfs": { ... },
  "sectors": [...],
  "us_stocks": { ... },
  "us_indexes": {
    "^DJI": {...},
    "^IXIC": {...},
    "^GSPC": {...}
  },
  "cn_indexes": {          // 新增：A股指数
    "000001.SS": {"name": "上证指数", "reason": "A股大盘"},
```

**变更**: 添加 `version` + `cn_indexes` + `cache`。缓存字段让非交易时段查询无需请求 yfinance。

### 5.5 validation_log.json（新增）

```json
{
  "version": 1,
  "entries": [
    {
      "date": "2026-07-09",
      "items": [
        {
          "stock": "300502",
          "name": "新易盛",
          "qa_view": "光模块调整接近尾声，1.6T放量在即",
          "qa_date": "2026.7.7",

          "verdict": "待观察",

        }

    }

```

### 5.6 RAG 增量 upsert

当前 `build_vector_db.py` 只支持全量重建。目标架构需要支持：

```bash
# 增/改单篇笔记（推荐，增量）
python scripts/build_vector_db.py --upsert 2026.7.9_光模块分析.md

# 改：笔记已变，更新向量
python scripts/build_vector_db.py --upsert 2026.7.9_光模块分析.md --force

# 删：笔记删除，删对应向量
python scripts/build_vector_db.py --remove 2026.7.9_光模块分析.md

# 重建（偶尔维护用）
python scripts/build_vector_db.py --rebuild
```

实现方式：用笔记文件名做 ChromaDB 的 ID prefix，增删改查通过 prefix 定位。

### 5.7 行情缓存机制

```
非交易时段（判断: 北京时间 15:00-21:30 + 周末）
  → 直接读 tracking_pool.cache，不请求 yfinance
  → 缓存标记 stale 条件: 超过 cache_ttl_minutes 或跨越交易日

交易时段
  → 请求 yfinance
  → 写 cache
  → 返回
```

## 六、模块职责与接口

### 6.1 模块地图

```
┌─────────────────────────────────────────────────────┐
│                 编排层（Hermes cron / CLI手动）       │
│  daily_brief.py 整合所有子模块输出                    │
└─────────────────────────────────────────────────────┘
         │                │                │
         ▼                ▼                ▼
┌─────────────┐  ┌──────────────┐  ┌──────────────────┐
│  数据采集     │  │   分析引擎    │  │   输出层          │
│             │  │              │  │                  │
│ tracking.py │  │ query_rag.py │  │ daily_brief.py  │
│ portfolio.py│  │ validate.py  │  │ 飞书webhook      │
│ transcribe  │  │              │  │ stdout           │
└─────────────┘  └──────────────┘  └──────────────────┘
```

### 6.2 接口规范

每个脚本必须支持:

```bash
# --help: 完整的用法说明
# --config: 指定 config.yaml 路径（默认自动寻找）
# --verbose / -v: 详细输出
```

### 6.3 模块解耦规则

| 规则 | 说明 | 违反示例 |
|------|------|---------|
| 查询引擎不写文件 | `query_rag.py` 只读，不修改任何数据文件 | query_rag 里写 validation_log |
| 数据模块不 import 其他业务模块 | `portfolio.py` 不 import `query_rag.py` | portfolio 里做 RAG 查询 |
| 只有编排层 import 多个模块 | `daily_brief.py` 可以同时用 portfolio 和 query_rag | 不允许 portfolio 直接调 tracking |
| 配置只从 config.yaml 读 | 不 `Path(__file__).parent.parent` | 所有路径走配置 |

## 七、演进路线图

### Phase 0 — 止血（当前，1-2天）

| 任务 | 优先级 | 描述 |
|------|--------|------|
| 创建 config.yaml | P0 | 统一配置层，所有路径走配置 |
| 添加 requirements.txt | P1 | 锁定依赖版本 |
| 数据文件加 version 字段 | P2 | 用 `migrate.py` 给现有文件加版本号 |
| 写本文档 | P0 | ✅ 已完成 |

### Phase 1 — 数据集中（1-2天）

| 任务 | 描述 |
|------|------|
| 创建 `data/` 目录，搬迁 portfolio/tracking 数据 | 脚本不变，只改 `config.yaml` 的数据路径 |

## 八、向目标架构迁移

### 8.1 不需要立即改的（保持兼容）

以下变更不需要一次性完成，新旧版本可以并存：

- `portfolio.json` 加 version 字段即使没有 v2 的全部字段也不影响运行
- `config.yaml` 缺失时脚本使用硬编码默认值
- 脚本同时支持 `config.yaml` 路径和 `Path(__file__).parent` 老路径

### 8.2 立即动手的（P0）

1. **创建 `config.yaml`** — 把 `E:/obsidian-vault/学习/钱博士` 等路径写进去
2. **创建 `data/` 目录** — 把 portfolio.json 和 tracking_pool.json 复制一份
3. **所有脚本加 `load_config()` 函数** — 从 config.yaml 读路径，不存在时回退默认

### 8.3 迁移策略

```
旧世界（当前）              过渡期                         新世界（目标）
─────────                   ─────                          ──────────
技能目录下的数据       ←→   config.yaml 指向旧位置   ←→   data/ 目录
Path(__file__).parent ←→   load_config() 有回退     ←→   配置层全接管
无版本号JSON           ←→   migrate.py 原地升级      ←→   v2+ 格式
全量重建RAG            ←→   --rebuild 仍可用          ←→   增量 upsert
```

过渡期策略:
- 旧文件不删除。`migrate.py` 在旧位置加 version 字段，原地升级
- 新脚本写新位置。旧脚本仍然能读旧位置
- 等所有脚本都更新到走 config.yaml 后，再物理搬迁

### 8.4 具体迁移步骤

```bash
# 1. 创建 data/ 目录
mkdir -p data/

# 2. 迁移现有数据（新位置，添加 version 字段）
python scripts/migrate.py portfolio   # 复制 + 升级 portfolio.json
python scripts/migrate.py tracking    # 复制 + 升级 tracking_pool.json
python scripts/migrate.py trades      # 创建 trades.json v1（如果不存在）

# 3. 配置 config.yaml
cp config.example.yaml config.yaml
# 编辑 config.yaml 中的路径

# 4. 验证
python scripts/portfolio.py status          # 自动检测 config.yaml
python scripts/tracking.py status           # 自动检测 config.yaml
python scripts/daily_brief.py --force       # 走新配置
```
